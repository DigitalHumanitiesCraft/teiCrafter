About teiCrafter

teiCrafter is a browser editor for TEI-XML editions. It opens an existing edition, shows the text page by page next to its facsimile, and lets you correct the text and annotate persons, places, organisations and works with authority identifiers (GND, GeoNames, Wikidata). It is a static page that runs entirely in the browser; the edition is read from and saved to a local file on your computer and is not uploaded anywhere.

What it is for, and what it is not

teiCrafter is a working environment for correcting and annotating TEI editions in the browser. Most editions reach it as the output of OCR or HTR pipelines that a person now verifies and refines. Plaintext files in a project folder open as minimal TEI drafted by a fixed rule (the text is carried verbatim); a separate, optional entry that drafts TEI from plain text with a language model exists and is currently switched off. It is not an XML development environment: there is no schema design and no transformation toolchain. Because saving changes nothing outside your deliberate edits, the same files can move between teiCrafter and tools such as oXygen or ediarum without one disturbing the other's work. It does not perform text recognition (that is upstream work) and does not publish editions; publishing is a separate step with its own tools.

What saving does (lossless)

Most XML software opens a file by parsing it into an internal tree and saves by writing that tree back out. The rewritten text is almost never the original: attribute order, entity notation (&#xE4; versus ä), whitespace inside tags and line endings are silently normalised, because the tree never recorded them. teiCrafter never rebuilds the file. The opened text itself remains the document; an edit replaces exactly the edited passage inside it, and everything else is preserved byte for byte. For example, a file containing <w norm="sich" orig="ſich"> comes back with the attributes in exactly that order and spelling. The practical consequence: a file comparison after saving shows precisely the human intervention, and nothing else, which is what makes editorial changes reviewable and provable. The editor checks this while you work and reports it next to the text (the well-formed, lossless chip); in addition, a re-runnable test suite in the repository round-trips several hundred real edition files and compares the result to the input.

What the editing unit is

TEI encodes its own structure, so the editor takes the editing unit from the document instead of asking for configuration: word-level when the TEI tokenizes words (<w>), line-level when it marks only line breaks (<lb>/<l>). Any TEI file opens without setup. A project can add settings on top through a manifest file (teicrafter.project.json next to the edition): its name, an image resolver for the facsimile, the markup its guidelines allow (declared per document type, since one project can hold letters, life documents and other genres), its indices and views. The facsimile pane reads <facsimile>/<zone> and links text and image in both directions via @facs.

What can be annotated

Entities (persons, places, organisations, works, events) live in a <standOff> index inside the document; selected text is linked to them as <name ref> mentions, and authority identifiers attach to an entity as <idno> elements (several registers per entity are possible). Editorial notes are written as <note target> entries, and textual-critical markup wraps a passage in <unclear>, <del> or <add>, or marks a <gap/>. All of this stays inside the saving behaviour described above; nothing is stored outside the file.

What leaves the browser

The edition itself is never transmitted; it is read from and saved to a local file. Network requests occur for three things: page images are fetched from the repositories the edition points to (for the examples: GAMS and the IIIF service of the Austrian National Library); an authority lookup sends the search term you typed to the chosen register (lobid.org for the GND, wikidata.org, geonames.org); and the page itself loads fonts and the OpenSeadragon viewer from public CDNs. When the optional language-model entry is enabled, the text you paste there is sent to the provider you select, and the API key is held in memory for the session only.

Browser requirements

A current browser with ES6-module support is required. Saving in place uses the File System Access API and is available in Chromium-based browsers; everywhere else the editor offers the edited file as a download instead. Nothing is installed.

Machine-generated content

Content that a machine produced is marked violet and counted as unreviewed until a human verifies it; violet is used for machine-generated content and for nothing else. An optional entry that lets a language model draft an initial TEI from plaintext exists in the codebase and is switched off in this preview.

The example projects

The tool is developed against three real projects, not all of them editions in the narrow sense: the Wenzelsbibel (word-level; the real Codex 2759 with IIIF facsimile where present, a synthetic stand-in for the licence-restricted codex in the public preview), a Jeanne Hersch letter from the Zentralbibliothek Zürich pipeline (line-level, page scans), and a letter from Stefan Zweig Digital (line-level, facsimile via the GAMS repository).

Method and provenance

teiCrafter is developed with Promptotyping, an AI-assisted, knowledge-document-driven development method; the knowledge base, the recorded decisions and the re-runnable tests are part of the repository: github.com/DigitalHumanitiesCraft/teiCrafter.

Status, citation, licence

teiCrafter is a research preview: it is under active development, its behaviour and interface change between versions, and it is not intended for production use.
